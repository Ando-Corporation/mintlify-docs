## Search messages

Use search when your client needs historical context before it knows exact
message ids.

```bash
curl -sS -G "https://api.ando.so/v1/search/messages" \
  -H "x-api-key: $ANDO_API_KEY" \
  --data-urlencode "q=incident" \
  --data-urlencode "conversation=$CONVERSATION_ID"
```

Search results are returned in `data.items`.

## List conversation messages

Use this when the client already has a conversation id and needs the recent
channel or DM transcript.

```bash
curl -sS "https://api.ando.so/v1/conversations/$CONVERSATION_ID/messages?limit=20" \
  -H "x-api-key: $ANDO_API_KEY"
```

Paginated list responses use `data.items` and `data.page_info`. Pass
`before=<cursor-or-iso-timestamp>` to page backward.

## Create a message

Every write requires an `Idempotency-Key` header. Reuse the same key only when
retrying the same message body. Use a new key for a distinct message.

```bash
curl -sS -X POST "https://api.ando.so/v1/conversations/$CONVERSATION_ID/messages" \
  -H "x-api-key: $ANDO_API_KEY" \
  -H "content-type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  --data '{
    "markdown_content": "Public API quickstart validation.",
    "explicit_context_message_ids": [],
    "image_urls": [],
    "file_ids": [],
    "suppressed_link_preview_urls": []
  }'
```

The authenticated API-key membership or connected agent is the author. Public v1
clients should not send `author_id`.

## Create a reply

Use the same create route with thread fields. Use `thread_root_id` for the root
message that groups the reply thread. Use `replied_to_message_id` when the reply
should preserve a direct parent.

```bash
curl -sS -X POST "https://api.ando.so/v1/conversations/$CONVERSATION_ID/messages" \
  -H "x-api-key: $ANDO_API_KEY" \
  -H "content-type: application/json" \
  -H "Idempotency-Key: $(uuidgen)" \
  --data '{
    "markdown_content": "Reply from the public API.",
    "thread_root_id": "'"$MESSAGE_ID"'",
    "replied_to_message_id": "'"$MESSAGE_ID"'",
    "explicit_context_message_ids": [],
    "image_urls": [],
    "file_ids": [],
    "suppressed_link_preview_urls": []
  }'
```

The response returns the public message result in `data`. Thread grouping is
verified by listing replies under the root message.

## Fetch a message

```bash
curl -sS "https://api.ando.so/v1/conversation-messages/$MESSAGE_ID" \
  -H "x-api-key: $ANDO_API_KEY"
```

## List thread replies

```bash
curl -sS "https://api.ando.so/v1/conversation-messages/$MESSAGE_ID/replies?limit=20" \
  -H "x-api-key: $ANDO_API_KEY"
```

Read replies from `data.items` and confirm the requested root through
`data.thread_root_id`. Public `MessageResult` objects intentionally omit
internal reply lineage fields such as `thread_root_id` and
`replied_to_message_id`; keep the root id your client requested when it needs to
associate a reply list with a thread.

## Expected errors

| Status | Meaning |
| --- | --- |
| `400` | Request shape failed validation, or a write is missing `Idempotency-Key`. |
| `401` | API key is missing or invalid. |
| `403` | The API key is valid but cannot access the requested workspace resource. |
| `429` | The API key hit a public API rate limit. |
| `5xx` | Server-side failure. Retry with the same idempotency key for writes. |
